# Linka 开发计划书

## 1. 项目结论

Linka 是一款极致轻量的 AI 驱动书签管理器，核心定位是：

> Save it. Let AI organize it.

中文表达：

> 只管收藏，AI 自动归档。

Linka 不做传统意义上的服务器 Dashboard，也不做复杂知识库系统。第一版只解决一个高频问题：用户平时看到有价值的网站、文章、工具、资料时，可以快速丢给 Linka，由 Linka 自动完成抓取、摘要、分类、标签、去重和卡片展示。

第一阶段目标是做出可在本地和群晖 NAS 上通过 Docker 部署的可用 MVP。

## 2. 产品定位

### 2.1 目标用户

- 有自托管 NAS 或家庭服务器的个人用户。
- 经常收藏网页、工具站、新闻、资料、教程、项目仓库的开发者或内容整理者。
- 希望减少手动分类和维护成本，只想快速保存链接的人。
- 希望通过自然语言找回已收藏内容的人。

### 2.2 核心价值

- 只需保存链接，不需要手动填写标题、分类和标签。
- AI 自动理解网页内容，生成摘要、分类和标签。
- 以卡片形式展示收藏内容，适合作为浏览器首页或个人资料入口。
- 通过 AI 助手完成收藏、搜索、整理和问答。
- 可用 Docker 部署在自己的群晖 NAS 上，数据自己掌控。

### 2.3 与同类项目的差异

| 项目 | 定位 | Linka 的差异 |
| --- | --- | --- |
| Heimdall | 服务器应用入口 Dashboard | Linka 更偏网页收藏和资料管理，不只是应用入口 |
| Homepage | 配置化自托管服务面板 | Linka 不要求手写 YAML，重点是 AI 自动整理 |
| Homarr | 拖拽式服务 Dashboard | Linka 更轻量，核心是收藏、摘要、搜索和归档 |
| Flame | 轻量 Startpage | Linka 在轻量卡片基础上加入 AI 助手和自动分类 |

## 3. 第一版范围

### 3.1 MVP 必做功能

1. 首页卡片展示
   - 以卡片形式展示已收藏链接。
   - 显示标题、域名、摘要、分类、标签、favicon 和封面图。
   - 支持按分类和标签筛选。
   - 支持关键词搜索。

2. 手动收藏链接
   - 用户输入 URL 后提交。
   - 后端抓取网页 metadata。
   - 自动解析标题、描述、favicon 和封面图。
   - 调用 AI 生成摘要、分类和标签。
   - 写入数据库并刷新首页卡片。

3. AI 助手
   - 页面右侧或底部提供轻量对话框。
   - 支持用户输入“收藏这个链接 https://...”。
   - 支持用户询问“我收藏过哪些 Docker 面板？”。
   - 支持从收藏库中检索并返回相关卡片。

4. 数据持久化
   - 第一版使用 SQLite。
   - 数据文件挂载到 Docker volume，便于群晖 NAS 备份。

5. Docker 部署
   - 提供 `Dockerfile`。
   - 提供 `docker-compose.yml` 示例。
   - 支持通过环境变量配置 AI 服务、数据库路径和访问令牌。

6. 外部收藏 API
   - 预留给后续 Chrome 扩展使用。
   - 支持通过 API Token 添加收藏。

### 3.2 第一版暂不做

- 多用户权限系统。
- 浏览器插件的完整实现。
- 移动 App。
- RSS 自动订阅。
- 复杂知识库、笔记系统和长文编辑器。
- 团队协作。
- 复杂向量数据库部署。
- 自动定时抓新闻。

这些功能可以作为第二阶段或第三阶段扩展，第一版不扩大范围。

## 4. 推荐技术架构

### 4.1 总体结构

```text
浏览器
  |
  | 访问 Web UI / 调用 API
  v
Linka Web 应用
  |
  | 后端 API
  v
服务层
  |-- 链接抓取
  |-- AI 整理
  |-- 搜索
  |-- 收藏管理
  |
  v
SQLite 数据库
```

### 4.2 技术选型

建议第一版使用以下组合：

| 层级 | 技术 | 原因 |
| --- | --- | --- |
| 前端 | Vue 3 + Vite + TypeScript | 开发轻量，适合构建卡片首页、筛选面板和 AI 对话框 |
| 后端 | Node.js + Fastify + TypeScript | API 简洁、性能足够、与前端共享类型方便 |
| 数据库 | SQLite | 群晖部署简单，单用户场景足够可靠 |
| ORM | Drizzle ORM 或 Prisma | 降低 SQL 维护成本，便于后续迁移 |
| AI 接入 | OpenAI 兼容 API | 支持 OpenAI，也支持兼容接口 |
| 部署 | Docker Compose | 适合群晖 NAS 和家庭服务器 |

### 4.3 低耦合原则

第一版也要保持基本分层：

- UI 层只负责展示和交互。
- API 层只负责请求响应和权限校验。
- 收藏服务负责新增、更新、删除和查询收藏。
- 抓取服务负责 URL 解析、metadata 抓取和正文提取。
- AI 服务负责摘要、分类、标签和对话意图识别。
- 数据层负责 SQLite 读写。

这样后续接 Chrome 扩展、浏览器分享入口或移动端时，不需要重写核心逻辑。

## 5. 数据模型

### 5.1 Bookmark

```text
Bookmark
- id
- url
- normalized_url
- title
- description
- summary
- domain
- favicon_url
- cover_image_url
- category_id
- pinned
- archived
- source
- created_at
- updated_at
```

字段说明：

- `url`：用户原始提交的链接。
- `normalized_url`：归一化后的链接，用于去重。
- `summary`：AI 生成的摘要。
- `category_id`：所属分类。
- `source`：来源，例如 `web`、`assistant`、`chrome-extension`。
- `pinned`：是否置顶。
- `archived`：是否归档。

### 5.2 Category

```text
Category
- id
- name
- slug
- color
- sort_order
- created_at
- updated_at
```

### 5.3 Tag

```text
Tag
- id
- name
- slug
- created_at
- updated_at
```

### 5.4 BookmarkTag

```text
BookmarkTag
- bookmark_id
- tag_id
```

### 5.5 ConversationMessage

```text
ConversationMessage
- id
- role
- content
- created_at
```

第一版可以只保留最近对话记录，不做复杂会话管理。

### 5.6 ApiToken

```text
ApiToken
- id
- name
- token_hash
- last_used_at
- created_at
- revoked_at
```

用于 Chrome 扩展和外部客户端调用收藏 API。

## 6. 核心 API 设计

### 6.1 收藏管理

```http
GET /api/bookmarks
```

查询收藏列表。

支持参数：

- `q`：关键词。
- `category`：分类。
- `tag`：标签。
- `archived`：是否查询归档内容。

```http
POST /api/bookmarks
```

新增收藏。

请求示例：

```json
{
  "url": "https://example.com/article",
  "title": "可选标题",
  "source": "web"
}
```

响应示例：

```json
{
  "status": "saved",
  "bookmark": {
    "id": "bookmark_id",
    "url": "https://example.com/article",
    "title": "示例文章",
    "summary": "这是一段由 AI 生成的摘要。",
    "category": "开发工具",
    "tags": ["工具", "效率"]
  }
}
```

```http
PATCH /api/bookmarks/:id
```

更新收藏，例如标题、分类、标签、置顶和归档状态。

```http
DELETE /api/bookmarks/:id
```

删除收藏。

```http
GET /api/bookmarks/check?url=...
```

检查链接是否已经收藏。该接口主要预留给 Chrome 扩展。

### 6.2 AI 助手

```http
POST /api/assistant/chat
```

请求示例：

```json
{
  "message": "帮我收藏这个网站 https://example.com"
}
```

响应示例：

```json
{
  "type": "bookmark_saved",
  "message": "已收藏，并归类到开发工具。",
  "bookmarkId": "bookmark_id"
}
```

助手需要支持的第一批意图：

- 收藏链接。
- 搜索已收藏内容。
- 查询某个分类下的收藏。
- 总结某个已收藏网页。

### 6.3 系统配置

```http
GET /api/settings
PATCH /api/settings
```

第一版可以只支持基础配置，例如默认分类、AI 模型名称和界面偏好。

## 7. AI 工作流

### 7.1 添加收藏流程

```text
用户提交 URL
  |
  v
校验 URL
  |
  v
归一化 URL 并检查重复
  |
  v
抓取网页 metadata 和正文摘要
  |
  v
调用 AI 生成标题建议、摘要、分类和标签
  |
  v
写入 SQLite
  |
  v
返回卡片数据
```

### 7.2 AI 输出结构

AI 整理链接时必须返回结构化 JSON：

```json
{
  "title": "网页标题",
  "summary": "一句到三句话的中文摘要",
  "category": "开发工具",
  "tags": ["AI", "效率", "工具"],
  "confidence": 0.86
}
```

### 7.3 提示词原则

- 只输出 JSON，不输出解释性文字。
- 分类名称保持简洁。
- 标签数量控制在 3 到 6 个。
- 摘要默认使用简体中文。
- 不确定时使用“未分类”，不要编造网页内容。

### 7.4 搜索策略

第一版采用混合搜索：

- 标题、摘要、描述、域名和标签做普通关键词匹配。
- AI 助手把自然语言问题转成关键词和分类过滤条件。

后续再升级为向量检索，避免第一版引入额外部署复杂度。

## 9. Docker 部署规划

### 9.1 环境变量

```env
LINKA_PORT=3030
LINKA_DB_PATH=/app/data/linka.sqlite
LINKA_APP_URL=http://localhost:3030

OPENAI_API_KEY=替换为实际密钥
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-4.1-mini

LINKA_API_TOKEN=替换为外部客户端访问令牌
```

### 9.2 docker-compose 示例

```yaml
services:
  linka:
    image: linka:latest
    container_name: linka
    restart: unless-stopped
    ports:
      - "3030:3030"
    volumes:
      - ./data:/app/data
    environment:
      - LINKA_PORT=3030
      - LINKA_DB_PATH=/app/data/linka.sqlite
      - OPENAI_API_KEY=${OPENAI_API_KEY}
      - OPENAI_BASE_URL=${OPENAI_BASE_URL}
      - OPENAI_MODEL=${OPENAI_MODEL}
      - LINKA_API_TOKEN=${LINKA_API_TOKEN}
```

### 9.3 群晖部署要求

- 支持 Container Manager 或 Docker。
- 将 `/app/data` 映射到群晖共享目录。
- 如果需要公网访问，建议通过群晖反向代理或 Nginx Proxy Manager 配置 HTTPS。
- 不建议将 OpenAI API Key 放到浏览器或 Chrome 扩展中。

## 10. Chrome 扩展预留方案

Chrome 扩展放到第二阶段开发。

### 10.1 扩展能力

- 点击工具栏按钮，一键收藏当前页面。
- 右键菜单 `Save to Linka`。
- 可选快捷键。
- 配置 Linka 服务地址和 API Token。
- 显示保存成功、已存在或失败状态。

### 10.2 扩展调用流程

```text
用户点击扩展按钮
  |
  v
扩展读取当前 tab 的 URL 和标题
  |
  v
调用 Linka 的 POST /api/bookmarks
  |
  v
Linka 后端完成抓取和 AI 整理
  |
  v
扩展展示保存结果
```

### 10.3 安全原则

- Chrome 扩展只保存 Linka API Token。
- Chrome 扩展不保存 OpenAI API Key。
- AI 调用只在 Linka 后端完成。
- API Token 支持撤销和重新生成。

## 11. 开发里程碑

### 11.1 阶段一：可运行骨架

目标：项目可以本地启动，并有基础页面和 API。

任务：

- 初始化前端和后端工程。
- 建立 SQLite 数据库。
- 实现基础布局。
- 实现 `GET /api/health`。
- 完成 Dockerfile 和本地启动脚本。

验收标准：

- 本地可以打开 Linka 首页。
- Docker 容器可以启动。
- 健康检查接口返回正常。

### 11.2 阶段二：收藏闭环

目标：用户可以添加链接并在首页看到卡片。

任务：

- 实现 `POST /api/bookmarks`。
- 实现网页 metadata 抓取。
- 实现 URL 去重。
- 实现收藏列表查询。
- 实现卡片网格展示。

验收标准：

- 输入 URL 后可以生成收藏卡片。
- 重复 URL 不会重复创建。
- 刷新页面后数据仍然存在。

### 11.3 阶段三：AI 整理

目标：新增收藏时自动生成摘要、分类和标签。

任务：

- 接入 OpenAI 兼容 API。
- 编写 AI 整理提示词。
- 实现结构化 JSON 解析和容错。
- 将分类和标签写入数据库。
- 在卡片中展示 AI 摘要和标签。

验收标准：

- 新收藏可以自动生成摘要。
- 分类和标签能正常展示。
- AI 失败时仍能保存基础收藏，不阻塞用户。

### 11.4 阶段四：AI 助手

目标：用户可以通过对话收藏和搜索。

任务：

- 实现助手 UI。
- 实现 `POST /api/assistant/chat`。
- 识别“收藏链接”意图。
- 识别“搜索收藏”意图。
- 返回相关收藏卡片。

验收标准：

- 输入“收藏这个链接 URL”可以成功新增收藏。
- 输入自然语言问题可以返回相关收藏。

### 11.5 阶段五：NAS 部署打磨

目标：可以稳定部署到群晖 NAS。

任务：

- 完善 Docker 镜像。
- 完善 `docker-compose.yml`。
- 编写部署文档。
- 确认数据目录持久化。
- 增加基础错误日志。

验收标准：

- 群晖 Docker 可以启动 Linka。
- 重启容器后数据不丢失。
- 环境变量配置生效。

### 11.6 阶段六：Chrome 扩展

目标：实现一键收藏当前网站。

任务：

- 创建 Manifest V3 扩展。
- 实现服务地址和 API Token 配置。
- 实现当前页面收藏。
- 实现右键菜单收藏。
- 显示保存结果。

验收标准：

- 在 Chrome 中点击扩展按钮可以收藏当前网页。
- Linka 首页能看到扩展添加的卡片。

## 12. 风险与处理策略

### 12.1 网页抓取失败

风险：

- 某些网站屏蔽抓取。
- 某些页面 metadata 不完整。
- 动态网页无法直接抓正文。

处理：

- 第一版只保证 metadata 抓取。
- 抓取失败时仍保存 URL 和用户提供的标题。
- 后续再考虑 Readability 或浏览器渲染抓取。

### 12.2 AI 输出不稳定

风险：

- AI 返回非 JSON。
- 分类过多或标签不稳定。
- 摘要出现不准确内容。

处理：

- 强制要求 JSON Schema。
- 解析失败时使用兜底分类“未分类”。
- 保留用户手动编辑能力。

### 12.3 群晖部署环境差异

风险：

- 文件权限问题。
- 网络代理问题。
- 反向代理路径问题。

处理：

- 数据目录统一挂载到 `/app/data`。
- 文档中明确权限和环境变量。
- 第一版优先支持根路径部署，不优先支持子路径部署。

### 12.4 API Token 泄露

风险：

- Chrome 扩展或浏览器配置中保存的 Token 被泄露。

处理：

- Token 只允许访问收藏相关 API。
- 支持撤销 Token。
- 不把 AI API Key 下发到客户端。

## 13. 第一版完成标准

第一版完成后，Linka 应满足：

- 可以通过 Docker Compose 启动。
- 可以在浏览器访问首页。
- 可以手动添加链接。
- 可以自动生成摘要、分类和标签。
- 可以以卡片形式展示收藏。
- 可以搜索和筛选收藏。
- 可以通过 AI 助手添加链接和查找收藏。
- 数据持久化到 SQLite。
- 为 Chrome 扩展预留外部 API 和 API Token 机制。
